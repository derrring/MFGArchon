- **`LagrangianBase.optimal_control` now returns the same alpha* as `HamiltonianBase.optimal_control`** (Issue #1642, capability B5). The base returned the bare conjugate maximizer `argmax_alpha {p.alpha - L}` -- i.e. `+p/lambda` under MINIMIZE -- while its own docstring promised "same alpha* as `HamiltonianBase.optimal_control()`", which returns `-p/lambda`, and while the analytic `SeparableLagrangian.optimal_control` override on the same class hierarchy also returns `-p/lambda`. Any `LagrangianBase` subclass that did not override the method therefore got a drift of the wrong sign, on a documented public subclassing path. `optimal_control` is now `-sign * dH/dp`: flipped under MINIMIZE, unchanged under MAXIMIZE (where the old code was accidentally right, since `sign = -1` cancels).
- `LagrangianBase.evaluate_hamiltonian` was **correct only because it compounded that error with a second one**: it evaluated `p.alpha* - L(alpha*)` at the wrong alpha*, and the two sign errors cancelled to the right `H`. Fixing either site alone would have shipped a regression -- under MINIMIZE the composed `DualLagrangian.evaluate_hamiltonian` would have returned `-H` for symmetric costs, and for an asymmetric `L` neither `L*(p)` nor `L*(-p)`. Both sites were fixed together and now share one owner, the new public `LagrangianBase.conjugate_argmax`, so the two consumers cannot re-fork into opposite conventions. `evaluate_hamiltonian` remains the convex conjugate `L*(p)`, sense-independent, matching `DualHamiltonian.__call__` and `ControlCostBase.evaluate` (Issue #1185).
- Measured on a harness covering every path that reaches either method (`DualLagrangian` over Quadratic/L1/Bounded x MINIMIZE/MAXIMIZE x V=f=0/V,f!=0, a plain non-overriding subclass, an asymmetric `L`, and the nD branch): **every** `evaluate_hamiltonian` value is byte-identical before and after, and the only changed values are the 34 MINIMIZE `optimal_control` rows, each an exact sign flip with the magnitude unchanged to the last printed digit. Pinning tests in `tests/unit/test_core/test_lagrangian_base_alpha_sign_1642.py` fail on a revert of either site individually and on a revert of both together.
- **`LagrangianBase.conjugate_argmax` is the public extension point for analytic subclasses** (breaking, pre-1.0, no shim). Because `evaluate_hamiltonian` reads the maximizer directly rather than going through `optimal_control`, a subclass supplying only an analytic `optimal_control` override -- the shape the old docstring advertised -- would have had its formula ignored by `evaluate_hamiltonian` and been routed through the numerical fallback instead. With `control_bounds() -> None` and `L(a) = 0.5a^2`, that returned `949.999` for an exact `5000.0` at `p=100` (81% low) with no warning. Overriding `conjugate_argmax` now supplies the closed form to both consumers at once; `optimal_control`'s docstring points there.
- **The `(-10.0, 10.0)` fallback search box no longer truncates silently.** When `control_bounds()` returns `None` the box is a solver stand-in, not the control set, so a maximizer terminating on its edge means the answer was truncated rather than optimized. `conjugate_argmax` and `proximal` now raise `ValueError` naming `control_bounds()` instead of returning the truncated value. Bounds that are genuinely *declared* are untouched -- an active constraint on a real control set is legitimate, and the guard keys on `control_bounds() is None`, not on the numbers, so a model declaring exactly `(-10.0, 10.0)` still returns its boundary optimum. The two routines now resolve the box through one owner, `_resolve_search_bounds`.
