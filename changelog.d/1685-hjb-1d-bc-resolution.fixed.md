- **1-D HJB silently substituted Neumann-0 for every faceless boundary condition** (Issue #1685). `_get_bc_info_1d` (`alg/numerical/hjb_solvers/base_hjb.py`) resolves the BC for the 1-D HJB FDM path. Its "no matching segment" branch read `bc.default_type` and its unified-interface branch read `bc.get_boundary_type()` -- **neither attribute exists**; the real names are `default_bc` and `get_bc_type_at_boundary`. Both raised `AttributeError` into a bare `except: pass`, so control fell through to a hardcoded `return BCType.NEUMANN, 0.0`. Every BC built by the `uniform_bc()` family -- which is every BC whose segments carry no `face`, including `dirichlet_bc(value=7.0)`, `neumann_bc(value=5.0)` and `periodic_bc()` -- resolved to zero-Neumann, and the solve ran on boundary data the caller never asked for, with no warning. Measured before: `dirichlet_bc(value=7.0)` and `no_flux_bc()` produced **byte-identical** `U`. After: they differ by `3.5`, and `periodic_bc()` differs from `no_flux_bc()` by `8.4e-01` on an asymmetric fixture. The exception handlers are gone along with the misspellings -- a `BoundaryConditions` that cannot answer these is a defect, not a fallback case.
- **A faced `NO_FLUX` segment crashed the same path** (Issue #1685). `NO_FLUX` fell through the enforcement branch's `if/elif` chain to `else: raise ValueError("Unsupported BC type ...")`, reachable from Auto Mode `problem.solve()` with no `scheme=` argument, from `fdm_upwind` and from `fdm_centered` -- while `HJBFDMSolver._SUPPORTED_BC_TYPES` declares `NO_FLUX` supported and the #1456 capability gate passes it. On the HJB side `NO_FLUX` **is** `du/dn = 0`, i.e. Neumann with a zero value (see `docs/user/guides/boundary_conditions.md`); it is a distinct condition only on the FP side, where it means zero total flux `J.n = 0`. It is now accepted by the Neumann branch on both boundaries.
- Both the type and the value are now read through `get_bc_type_at_boundary` / `get_bc_value_at_boundary`. A first version of this fix read the value through the accessor but the type from the raw `bc.default_bc` attribute, which is independent of the segments: a faceless `DIRICHLET` segment with `value=9.0` beside `default_bc=NEUMANN` resolved to `(NEUMANN, 9.0)`, applying `du/dn = 9.0` where the caller declared `u = 9.0`. `default_bc` is also `BCType | None`, so an unset default resolved to a `None` bc_type, and reading it raw bypasses `_resolve_default_bc` (the Issue #1100 diagnostic). `tests/unit/test_alg/test_hjb_bc_resolution_1685.py` pins both: restoring the original defect fails 9 of its 10 cases.
- Robin is **not** among the types this fixes. `HJBFDMSolver`'s Issue #1456 capability gate rejects `BCType.ROBIN` before the resolver is reached, and the Priority-2 path would still return the default `alpha`/`beta` rather than the segment's, so a faceless Robin BC would degenerate to Dirichlet if it could get there. Recorded rather than fixed, since it is unreachable.
