- **The nightly full-suite backstop now runs to completion and reports its failures** (Issue #1658). It had **never** completed: 0 successes in 108 runs between 2026-04-03 and 2026-07-19 (98 cancelled at the 45-minute wall, 10 aborted by `--maxfail=10` at roughly 2.5% of the suite). Five causes, all fixed: it ran **serially** with no `-n auto`, unlike `ci.yml` and `scripts/local_ci.sh`; it ran **unsharded**, and the `@slow` subset alone measures 32m33s locally under `-n auto`, so removing `--maxfail` alone would only have traded an early abort for a timeout cancellation -- it is now a four-way matrix (unit / integration / validation / regression) with `fail-fast: false`, so wall-clock is the slowest shard and one red shard cannot cancel the others; `--maxfail=10` stopped it before `tests/unit/` was ever reached, so **every `@slow` test in the repository was unexecuted**, not merely a subset; it installed a `.[dev,numerical]` extra that `pyproject.toml` never declared, so `cvxpy` was absent and the SOCP gate burned a `maxfail` slot on `ImportError` every night; and nothing reported a failure anywhere, so 3.5 months of red went unnoticed. A separate `notify-on-failure` job now opens an issue (de-duplicated on title, commenting on the existing one instead of opening a new one per night), modelled on the pattern already used in `docs_audit.yml`. It is gated `always() && needs.full-test-suite.result != 'success'` rather than `failure()`, because timeout exhaustion is reported as `cancelled` and `failure()` is false for a cancelled dependency -- which was 98 of 108 historical runs. A per-shard step would also have raced: three concurrent failures would each create an issue instead of one. `tests/unit/test_backends/` and `tests/unit/test_visualization/` remain `--ignore`d by an explicit deferral (they run in no AUTOMATIC workflow -- the release-only `pytest tests/` at `ci.yml:190-194` has no `--ignore`, so they do execute there; tracked in #1659), now annotated as such in the workflow rather than left as a bare flag. The `numerical` extra is now declared and included in `all`, so `pip install mfgarchon[all]` no longer omits a dependency a shipped feature requires. The review of the first attempt caught two further defects: the matrix listed three of `tests/`'s **four** directories, silently dropping `tests/regression/` -- which holds `test_solver_golden_baseline.py`, the `atol=1e-9, rtol=0` byte-identity gate and the one automatic detector of this repo's dominant bug class -- and the notification ran per-shard, so three concurrent failures would each have created an issue rather than one. Notification is now a single dependent job with pagination, and is deliberately not gated on `schedule` so the path can be exercised by `workflow_dispatch` before it is needed.
