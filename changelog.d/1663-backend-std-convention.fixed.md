- **Backends disagreed on what `std` means** (Issue #1663). `BaseBackend.std` documented only *"Compute standard deviation"* and never stated the convention, so each implementer inherited its library's default: NumPy and JAX returned the population value (`N`), `TorchBackend` returned the Bessel-corrected one (`N-1`). Measured on the same input: `0.314315110` against `0.312755227`, a ratio of exactly `sqrt(N/(N-1))`. The contract now specifies population (`ddof=0`) -- these arrays are discretised fields, not samples, so there is no sampling correction to make -- and `TorchBackend.std` passes `correction=0`. A new `tests/unit/test_backends/test_backend_reduction_conventions.py` pins agreement across every constructible backend for `mean`/`std`/`max`/`min`, and pins the convention itself separately, since three backends agreeing on the *wrong* convention would satisfy an agreement-only check.
- **The test that should have caught it could not** (Issue #1663). `test_cross_backend_consistency.py::test_backend_factory_integration` called `backend.sum()` -- a method that has never been in the contract, whose abstract reductions are `mean`/`std`/`max`/`min`/`trapezoid`. Every backend raised `AttributeError`, an `except Exception: print(...)` swallowed it, the results dict stayed empty, and the comparison was skipped as *"Only one backend available"* although all three had constructed successfully. The swallow is gone (a backend that errors now fails the test), the skip distinguishes "one backend" from "every backend produced nothing", and two latent flaws surfaced once it began running at all: it compared a **relative** error against a reference of `~0` (a full-period sine averages to zero, so two correct answers differed by a "relative error" of 2.3e7), and its `1e-6` tolerance was unreachable for a float32 backend -- on Apple Silicon the torch backend is forced to float32, with a warning, because MPS has no float64.
