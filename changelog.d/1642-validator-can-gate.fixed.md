- **`validate_hamiltonian_consistency` can now fail a problem instead of only warning** (Issue #1642, capability `C1`). The validator held only `add_warning`, so it could never set `is_valid` False and the `if not result.is_valid: raise` branch in `MFGProblem` was structurally dead for it -- a check that could not gate. Genuine inconsistencies now raise with the offending values named; heuristic observations remain warnings.
- **`validate_custom_functions(check_consistency=...)` now defaults to `True`** (Issue #1642). Behavior change for every caller: the derivative-consistency check runs unless explicitly disabled, and it can now fail construction. `MFGProblem` no longer passes `check_consistency=False`, so a wrong `dH_dm`/`dH_dp` is refused at construction instead of silently steering every Picard step. Pass `check_consistency=False` to restore the previous behavior.
- **Moreau-Yosida regularized control costs counted the smoothing penalty once per dimension** (Issue #1642). `_MoreauYosidaControlCost.evaluate` added a penalty summed over components to a base term that `L1ControlCost` and `BoundedControlCost` return *per component*, so `H_eps` overcounted the penalty by a factor of `d`: exact in 1-D, wrong for every `d > 1`, and inconsistent with `dH_eps/dp`. Batch input `(N, d)` now also reduces to `(N,)` as the `SeparableHamiltonian` / `CongestionHamiltonian` call sites assume. With the consistency check above now live, this defect made `MFGProblem` refuse any `regularize()`-smoothed cost in 2-D and 3-D.
