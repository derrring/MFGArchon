- **`ControlCost.evaluate` reduced over the component axis in one class and not the other two** (Issue #1672, D9). `QuadraticControlCost.evaluate` summed over `axis=-1`; `L1ControlCost` and `BoundedControlCost` returned one value per component. Every class's own `lagrangian` already summed, so the convention was settled and `evaluate` was the outlier -- this aligns it rather than choosing anything. Consequence, measured: with `L1` or `Bounded`, `SeparableHamiltonian` returned an **`(N, N)` matrix at `d=1`** -- silently, since a square array is a plausible shape for a Hamiltonian on a grid -- and raised a broadcast `ValueError` at `d != 1`. `_MoreauYosidaControlCost.evaluate` carried a local workaround (`np.sum(self.base.evaluate(q), axis=-1)`) written to paper over the fork; it is removed in the same change, since with a consistent base it would double-reduce and collapse the batch axis. Two existing tests pinned the old convention and are updated; one of them exposed a real ambiguity, that a bare `(d,)` array is one point with `d` components rather than `d` scalar points -- the reading `QuadraticControlCost` has always used -- so it now states shapes explicitly instead of relying on it.
- Two docstrings in `hamiltonian.py` corrected. `L1ControlCost`'s example is updated to the new total. **`QuadraticControlCost`'s was already wrong before this change**: it advertised `cost.evaluate(np.array([1.0, 2.0]))  # [0.25, 1.0]` for `lambda_=2.0`, while that class has always summed and returns `1.25`. A documented per-component output for the one class that never had one.
