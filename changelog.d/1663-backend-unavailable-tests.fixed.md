- **Three backend-unavailability tests asserted nothing** (Issue #1663). `test_create_backend_{torch,jax,numba}_when_unavailable` tried to simulate a missing backend by popping the name out of the module-level `_BACKENDS` dict, then skipped when the backend *was* installed -- so on a developer machine with torch present they never ran, and on one without it the pop was redundant. The pop also does not simulate anything: `create_backend` imports the backend module directly, so after `_BACKENDS.pop("torch")` it still returns a working `TorchBackend`. The tests now block the import itself via `monkeypatch.setitem(sys.modules, "mfgarchon.backends.torch_backend", None)`, which reaches the real code path, and the skips are gone -- 37 passed / 3 skipped becomes 40 passed. This surfaced a second defect: the torch case asserted `match="PyTorch is required for TorchBackend"`, a string `create_backend` never emits; the actual message is `"PyTorch backend requested but not available"`, so the assertion would have passed on any `ImportError` at all.
