- **Three backend-unavailability tests asserted nothing** (Issue #1663). `test_create_backend_{torch,jax,numba}_when_unavailable` each began with `pytest.skip` when the backend *was* installed, so on any machine that had it -- every developer machine and CI runner here -- the body never ran. Underneath, the simulation was wrong too: the tests popped the name out of `_BACKENDS`, but `create_backend` re-imports and re-registers on demand, so after `_BACKENDS.pop("torch")` it still returns a working `TorchBackend`. The skips are gone and all three now reach a real failure path: 37 passed / 3 skipped becomes 40 passed.
- **The three backends do not fail alike, and two of the tests were checking the wrong path** (Issue #1663). `torch_backend.py` and `jax_backend.py` import successfully without their library -- they degrade to `TORCH_AVAILABLE` / `JAX_AVAILABLE` flags -- so `backends/__init__.py` registers both into `_BACKENDS` regardless of whether the library exists. For those two the `if backend_name not in _BACKENDS` branch is unreachable on any install, and the error a real user sees is raised by the constructor: `"PyTorch is required for TorchBackend. Install with: pip install torch"` (`torch_backend.py:109`) and `"JAX backend requested but JAX is not installed..."` (`jax_backend.py:66`). The tests now clear the availability flag, which reproduces exactly that path; mutating either production string fails the corresponding test, where previously both could be changed freely with the suite still green. `numba_backend.py` is the exception -- it raises `ImportError` at import time and is never pre-registered -- so its lazy-registration branch is genuinely live and its test blocks the import instead. The asymmetry is recorded in the test docstrings so the three are not "made consistent" by someone who has not checked which paths exist.
