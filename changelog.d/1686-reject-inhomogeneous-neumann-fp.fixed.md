- **FP solvers now refuse a Neumann value they would have discarded** (Issue #1686). All six FP families declare `BCType.NEUMANN` in `_SUPPORTED_BC_TYPES` and then read only the type: `neumann_bc(value=g)` with `g != 0` is applied by the HJB side and silently dropped by the FP side, so the coupled solve integrates a pair that is not adjoint and still reports `converged=True`. Measured: on the FP side `max|M(g=0) - M(g=-100)| = 0.0` while the HJB side moved by `1.6e+03`; at `g=-100` roughly 90% of the mass should have left through the wall and none did. The #1456 capability gate could not catch it because it tests `{seg.bc_type}` for set membership and never inspects `seg.value`. A new per-solver flag `honors_inhomogeneous_neumann` (default `True`, set `False` by all six FP families) makes the gate refuse a non-zero `g` with a message naming the value. This is the honest half only -- **the inhomogeneous flux wall is not implemented**, and the correctness half remains open. `g = 0`, equivalently `no_flux_bc()`, is unaffected and is the whole of current usage, so nothing that worked stops working. The flag is per-solver rather than global because the same `BCType.NEUMANN` means `du/dn = g` on the HJB side and a prescribed flux `J.n = g` on the FP side (see `docs/user/guides/boundary_conditions.md`); the HJB side continues to accept it. A first version of the guard tested `isinstance(seg.value, (int, float))`, which let a callable through: `neumann_bc(value=lambda t: 5.0)` was accepted and discarded silently -- the exact behaviour the guard exists to stop, reintroduced by the guard. Time-dependent values are now refused too, since a callable cannot be shown identically zero without evaluating it everywhere; the message says that rather than claiming the value is non-zero, and points at `0.0` for anyone whose function is constant. The HJB side continues to accept `du/dn = g(t)`.
