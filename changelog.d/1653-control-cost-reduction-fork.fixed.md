- **`ControlCost.evaluate` reduced over the component axis in one class and not the other two** (Issue #1653). `QuadraticControlCost.evaluate` summed over `axis=-1`; `L1ControlCost` and `BoundedControlCost` returned one value per component. Every class's own `lagrangian` already summed, so the convention was settled and `evaluate` was the outlier -- this aligns it rather than choosing anything. Consequence, measured on the `SeparableHamiltonian` path with `L1` or `Bounded`: it returned an **`(N, N)` matrix wherever the componentwise array could broadcast against the per-point one** -- that is, whenever `d == 1` **or** `N == d` -- and raised a broadcast `ValueError` only when `N != d` and `d != 1`. Silently, since a square array is a plausible shape for a Hamiltonian on a grid. Review of this PR corrected an earlier claim here that the silent surface was `d = 1` alone; `N == d` is a second, and `evaluate` itself never raised at any arity -- the raise came from the caller. `_MoreauYosidaControlCost.evaluate` carried a local workaround (`np.sum(self.base.evaluate(q), axis=-1)`) written to paper over the fork; it is removed in the same change, since with a consistent base it would double-reduce and collapse the batch axis. Two existing tests pinned the old convention and are updated; one of them exposed a real ambiguity, that a bare `(d,)` array is one point with `d` components rather than `d` scalar points -- the reading `QuadraticControlCost` has always used -- so it now states shapes explicitly instead of relying on it.
- Two docstrings in `hamiltonian.py` corrected. `L1ControlCost`'s example is updated to the new total. **`QuadraticControlCost`'s was already wrong before this change**: it advertised `cost.evaluate(np.array([1.0, 2.0]))  # [0.25, 1.0]` for `lambda_=2.0`, while that class has always summed and returns `1.25`. A documented per-component output for the one class that never had one.
